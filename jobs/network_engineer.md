## 一、基础理论：必须烂熟于心

这部分是面试的"入场券"，答错基本就直接凉凉。

### 1. OSI 七层模型与 TCP/IP 四层模型

**高频题：请简述每层的功能及常见协议**

#### OSI 模型（从上到下）：

**应用层**
- 功能：为应用程序提供网络服务（文件传输、邮件、网页）
- 协议：HTTP、FTP、SMTP、DNS

**表示层**
- 功能：数据格式转换、加密解密、压缩解压
- 协议：SSL/TLS、JPEG、ASCII

**会话层**
- 功能：建立、管理和终止会话连接，控制通信节奏
- 协议：RPC、SQL、NFS

**传输层**
- 功能：提供端到端的可靠或不可靠的数据传输服务
- 协议：TCP（可靠，面向连接）、UDP（不可靠，无连接）

**网络层**
- 功能：负责数据包的路由选择和转发
- 协议：IP（IPv4/IPv6）、ICMP、IGMP、ARP、RIP、OSPF、BGP

**数据链路层**
- 功能：将比特流封装成帧，进行差错控制和流量控制
- 协议：Ethernet、PPP、HDLC、MAC

**物理层**
- 功能：传输原始比特流，定义电气、机械、功能和过程规范
- 协议：RJ45、光纤、无线电频率

#### TCP/IP 模型（从上到下）：

- 应用层（Application Layer）：对应于 OSI 参考模型的应用层、表示层、会话层
- 传输层（Transport Layer）：对应于 OSI 参考模型的传输层
- 网际层（Internet Layer）：对应于 OSI 参考模型的网络层
- 网络接口层（Network Interface Layer）：对应于 OSI 参考模型的数据链路层和物理层

#### 对比：为什么现在普遍使用 TCP/IP 而不是 OSI？

OSI理论完美但复杂、实现难、出现晚；TCP/IP简单实用、早成熟、互联网全靠它，所以成为实施标准

### 2. TCP/UDP 的核心机制

#### 三次握手与四次挥手：状态位变化及原理

**状态位说明**
- **SYN** (Synchronize): 同步序列号，建立连接时使用
- **ACK** (Acknowledgment): 确认标志，表示确认收到数据
- **FIN** (Finish): 结束标志，请求断开连接
- **RST** (Reset): 重置连接，异常情况下使用

**三次握手详细过程**

```
客户端                    服务器
   |                        |
   |----SYN=1,seq=x-------->|  (第一次握手)
   |                        |
   |<---SYN=1,ACK=1,seq=y---|  (第二次握手)
   |    ack=x+1             |
   |                        |
   |----ACK=1,seq=x+1------>|  (第三次握手)
   |    ack=y+1             |
   |________连接建立_______|
```

**为什么是三次握手而不是两次？**

核心原因：防止已失效的连接请求报文突然又传送到了服务器

1. 两次握手的问题：
   - 客户端发送连接请求后突然宕机重启
   - 之前的连接请求在网络中延迟到达服务器
   - 服务器误认为是新的连接请求，建立连接
   - 但客户端已经不需要这个连接，造成资源浪费

2. 三次握手的优势：
   - 客户端收到服务器的确认后，还要再发送一次确认
   - 服务器只有收到客户端的最终确认才建立连接
   - 确保双方都准备好进行数据传输

**四次挥手详细过程**

```
主动关闭方              被动关闭方
     |                      |
     |-----FIN=M----------->|  (第一次挥手)
     |                      |
     |<----ACK=M+1---------|  (第二次挥手)
     |                      |
     |<----FIN=N-----------|  (第三次挥手)
     |                      |
     |-----ACK=N+1-------->|  (第四次挥手)
     |______连接断开______|
```

**为什么挥手要四次？**

根本原因：TCP是全双工通信，需要分别关闭两个方向的连接

1. 第一次挥手：主动关闭方发送FIN，表示自己没有数据要发送了
2. 第二次挥手：被动关闭方回复ACK，确认收到关闭请求
3. 第三次挥手：被动关闭方发送FIN，表示自己也没有数据要发送了
4. 第四次挥手：主动关闭方回复ACK，连接完全关闭

### TCP状态变化对照图

```
┌─────────────┬─────────────────────────────────────────────────────────────┐
│   阶段      │                    客户端 ↔ 服务器                          │
├─────────────┼─────────────────────────────────────────────────────────────┤
│ 初始状态    │  CLOSED          ↔         CLOSED                           │
│ 监听状态    │  CLOSED          ↔         LISTEN    (服务器监听端口)        │
│ 第一次握手  │  SYN_SENT        ↔         SYN_RCVD  (收到SYN)              │
│ 连接建立    │  ESTABLISHED     ↔         ESTABLISHED                       │
│ 数据传输    │  ESTABLISHED     ↔         ESTABLISHED  (双向数据传输)        │
│ 主动关闭    │  FIN_WAIT_1      ↔         ESTABLISHED  (发送FIN)            │
│ 被动响应    │  FIN_WAIT_1      ↔         CLOSE_WAIT   (收到FIN,发送ACK)     │
│ 等待关闭    │  FIN_WAIT_2      ↔         CLOSE_WAIT   (等待对方关闭)        │
│ 被动关闭    │  FIN_WAIT_2      ↔         LAST_ACK     (发送FIN)            │
│ 最终确认    │  TIME_WAIT       ↔         CLOSED       (收到最后ACK)         │
│ 连接终止    │  CLOSED          ↔         CLOSED                            │
└─────────────┴─────────────────────────────────────────────────────────────┘
```

### 详细状态流转图

```
客户端状态流转图：
CLOSED → SYN_SENT → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
   ↑                                                    ↓
   └────────────────────────────────────────────────────┘

服务器状态流转图：
CLOSED → LISTEN → SYN_RCVD → ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
   ↑                                           ↓
   └───────────────────────────────────────────┘
```

### 状态对应关系表

| 时间点 | 客户端状态 | 服务器状态 | 说明 |
|--------|------------|------------|------|
| T0 | CLOSED | CLOSED | 初始状态 |
| T1 | SYN_SENT | LISTEN | 客户端发起连接请求 |
| T2 | SYN_SENT | SYN_RCVD | 服务器收到SYN |
| T3 | ESTABLISHED | ESTABLISHED | 连接建立完成 |
| T4 | ESTABLISHED | ESTABLISHED | 数据传输阶段 |
| T5 | FIN_WAIT_1 | ESTABLISHED | 客户端发起关闭 |
| T6 | FIN_WAIT_1 | CLOSE_WAIT | 服务器确认关闭请求 |
| T7 | FIN_WAIT_2 | CLOSE_WAIT | 客户端等待服务器关闭 |
| T8 | FIN_WAIT_2 | LAST_ACK | 服务器发起关闭 |
| T9 | TIME_WAIT | CLOSED | 客户端等待最后确认 |
| T10 | CLOSED | CLOSED | 连接完全关闭 |

### 关键状态说明

**TIME_WAIT状态的重要性**：
- 持续时间：2×MSL（通常2-4分钟）
- 作用：确保对方收到确认，且网络中所有迟到的报文都消失
- 防止：旧连接的报文干扰新连接

**CLOSE_WAIT状态**：
- 出现原因：被动关闭方未及时调用close()
- 解决方法：应用程序应及时关闭socket
- 影响：大量CLOSE_WAIT可能导致资源泄露

### 重要时间参数

- **MSL (Maximum Segment Lifetime)**: 报文最大生存时间，通常为2分钟
- **TIME_WAIT状态**: 持续2MSL时间，确保网络中所有报文都消失
- **作用**: 防止旧连接的报文干扰新连接

#### 可靠性保证机制

**1. 检验和（Checksum）**
- **作用机制**：
  - 错误检测：检测数据传输过程中的比特错误
  - 端到端校验：覆盖TCP头部和数据部分
  - 16位校验和：使用反码求和算法计算
  - 丢弃错误包：检测到错误时直接丢弃数据包

- **应用场景示例**：
  网络传输中发生电磁干扰导致数据包某位翻转：
  - 发送方计算：原始数据 + TCP头部 = 校验和
  - 接收方验证：收到数据 + TCP头部 ≠ 校验和
  - 自动丢弃：接收方发现校验失败，丢弃该数据包
  - 触发重传：发送方超时未收到确认，自动重传正确数据

**2. 序列号（Sequence Number）**
- **作用机制**：
  - 唯一标识：每个字节数据都有唯一的序列号
  - 数据重组：接收方按序列号重新排列乱序到达的数据包
  - 去重处理：识别并丢弃重复的数据包

- **应用场景示例**：
  当浏览器下载一个1MB的图片文件时：
  - 发送方为每个字节分配序列号：从1000开始递增
  - 第一个数据包：Seq=1000, Len=1460（包含1460字节数据）
  - 第二个数据包：Seq=2460, Len=1460
  - 如果网络中第二个包先到达，接收方根据序列号2460将其缓存，等待序列号1000的数据包到达后再按正确顺序组装

**3. 确认应答（Acknowledgment）**
- **作用机制**：
  - 累积确认：确认号表示之前所有数据都已正确接收
  - 选择确认(SACK)：可选择性确认非连续的数据块
  - 延迟确认：提高效率，减少ACK报文数量

- **应用场景示例**：
  Web服务器向客户端发送网页数据：
  - 服务器发送数据包：Seq=5000, Len=1000
  - 客户端收到后发送确认：Ack=6000（表示期望下一个序列号为6000的数据）
  - 采用延迟确认：客户端等待收到第二个数据包或200ms超时后才发送ACK，减少网络流量

**4. 超时重传（Retransmission Timeout）**
- **作用机制**：
  - 超时时间计算：RTO = RTTs + 4 × RTTd
  - 重传触发条件：超时重传、快速重传、早期重传
  - 重传算法优化：指数退避算法

- **应用场景示例**：
  视频会议中传输音频数据：
  - 发送方发送音频包：Seq=8000
  - 设置RTO=300ms等待确认
  - 300ms内未收到Ack=8000的确认，触发超时重传
  - 如果收到3个重复的Ack=8000，则立即快速重传（不等待超时）
  - 重传间隔按指数退避：第1次1秒，第2次2秒，第3次4秒

**5. 滑动窗口（Sliding Window）**
- **作用机制**：
  - 窗口机制原理：动态调整发送和接收窗口大小
  - 流量控制：防止发送方压垮接收方
  - 窗口控制参数：通告窗口(rwnd)、拥塞窗口(cwnd)

- **应用场景示例**：
  文件传输场景：
  - 发送方窗口大小：4个数据包
  - 同时发送4个包：Seq=1000, Seq=2000, Seq=3000, Seq=4000
  - 收到第一个确认后，窗口滑动，可以发送Seq=5000
  - 接收方通告窗口：rwnd=8192字节，表示还能接收8KB数据
  - 实际发送窗口=min(cwnd,rwnd)，实现动态流量控制

**6. 拥塞控制（Congestion Control）**
- **作用机制**：
  - 慢启动：初始阶段指数增长
  - 拥塞避免：线性增长阶段
  - 快速重传：收到3个重复ACK立即重传
  - 快速恢复：重传后不回到慢启动

- **应用场景示例**：
  大文件下载过程：
  - 连接建立初期：慢启动阶段，cwnd从1MSS开始指数增长
  - 达到阈值后：进入拥塞避免，线性增长避免网络过载
  - 检测到丢包：收到3个重复Ack=15000，触发快速重传Seq=15000
  - 重传后：cwnd减半，进入快速恢复阶段而非重新慢启动
  - 整个过程自动适应网络状况，平衡传输效率和网络稳定性

### 3. IP 地址与子网划分

#### IP地址分类

**IPv4地址分为五类，根据第一个八位组的数值范围划分：**

**A类地址（1.0.0.0 - 126.255.255.255）**
- **网络位**：第一个八位组（8位）
- **主机位**：后三个八位组（24位）
- **默认子网掩码**：255.0.0.0 或 /8
- **网络数量**：126个（2^7 - 2）
- **每网络主机数**：约1677万台（2^24 - 2）
- **典型用途**：大型企业、ISP骨干网络
- **私有地址范围**：10.0.0.0 - 10.255.255.255

**B类地址（128.0.0.0 - 191.255.255.255）**
- **网络位**：前两个八位组（16位）
- **主机位**：后两个八位组（16位）
- **默认子网掩码**：255.255.0.0 或 /16
- **网络数量**：16384个（2^14）
- **每网络主机数**：约6.5万台（2^16 - 2）
- **典型用途**：中型企业网络
- **私有地址范围**：172.16.0.0 - 172.31.255.255

**C类地址（192.0.0.0 - 223.255.255.255）**
- **网络位**：前三个八位组（24位）
- **主机位**：最后一个八位组（8位）
- **默认子网掩码**：255.255.255.0 或 /24
- **网络数量**：2097152个（2^21）
- **每网络主机数**：254台（2^8 - 2）
- **典型用途**：小型企业、家庭网络
- **私有地址范围**：192.168.0.0 - 192.168.255.255

**D类地址（224.0.0.0 - 239.255.255.255）**
- **用途**：组播地址
- **特点**：用于一对多通信
- **典型应用**：视频会议、在线直播、多媒体传输

**E类地址（240.0.0.0 - 255.255.255.255）**
- **用途**：实验和研究用途
- **特点**：保留地址，不用于常规网络通信

**特殊地址说明：**
- **网络地址**：主机位全为0，标识网络本身
- **广播地址**：主机位全为1，用于向本网段所有主机发送数据
- **回环地址**：127.0.0.0/8，用于本地测试
- **APIPA地址**：169.254.0.0/16，当DHCP失败时自动分配

#### 快速区分IP地址类别方法

**二进制首位判断法（最准确）**

观察IP地址第一个八位组的二进制表示：

```
A类地址：首位为0
0xxxxxxx.xxxxxxxx.xxxxxxxx.xxxxxxxx
范围：00000001 - 01111110 (1-126)

B类地址：前两位为10
10xxxxxx.xxxxxxxx.xxxxxxxx.xxxxxxxx
范围：10000000 - 10111111 (128-191)

C类地址：前三位为110
110xxxxx.xxxxxxxx.xxxxxxxx.xxxxxxxx
范围：11000000 - 11011111 (192-223)

D类地址：前四位为1110
1110xxxx.xxxxxxxx.xxxxxxxx.xxxxxxxx
范围：11100000 - 11101111 (224-239)

E类地址：前四位为1111
1111xxxx.xxxxxxxx.xxxxxxxx.xxxxxxxx
范围：11110000 - 11111111 (240-255)
```

#### IP计算题

**经典面试题：给定IP地址192.168.10.130/27，请计算：**

1. **网络地址**
2. **广播地址** 
3. **可用主机地址范围**
4. **可用主机数量**

**详细计算过程：**

**第一步：分析给定信息**
- IP地址：192.168.10.130
- 子网掩码：/27（CIDR表示法）
- 二进制子网掩码：11111111.11111111.11111111.11100000
- 点分十进制掩码：255.255.255.224

**第二步：确定网络位和主机位**
- 网络位：前27位（用1表示）
- 主机位：后5位（用0表示）
- 主机位数量：5位
- 可用主机数：2^5 - 2 = 32 - 2 = 30台（减去网络地址和广播地址）

**第三步：计算网络地址**
将IP地址与子网掩码进行按位与运算：

```
IP地址：  192.168.10.130 = 11000000.10101000.00001010.10000010
掩码：    255.255.255.224 = 11111111.11111111.11111111.11100000
------------------------------------------------------------
网络地址：192.168.10.128 = 11000000.10101000.00001010.10000000
```

**第四步：计算广播地址**
网络地址的主机位全部置1：

```
网络地址：192.168.10.128 = 11000000.10101000.00001010.10000000
广播地址：192.168.10.159 = 11000000.10101000.00001010.10011111
```

**第五步：确定可用主机范围**
- 起始地址：网络地址 + 1 = 192.168.10.129
- 结束地址：广播地址 - 1 = 192.168.10.158
- 可用主机范围：192.168.10.129 ~ 192.168.10.158

**第六步：验证计算结果**
- 网络地址：192.168.10.128
- 广播地址：192.168.10.159
- 可用主机数：30台
- 子网块大小：32个地址（2^5）

**快捷计算方法：**
对于/27子网：
- 子网增量：256 - 224 = 32
- 网络地址：找到小于等于130的最大32的倍数 = 128
- 广播地址：网络地址 + 31 = 128 + 31 = 159

#### IPv6相关

- **与 IPv4 的主要区别**：
  - 地址长度：IPv6为128位，IPv4为32位
  - 地址表示：IPv6使用十六进制，IPv4使用十进制
  - 地址空间：IPv6约3.4×10³⁸个地址，IPv4约43亿个地址
  - 报头结构：IPv6报头更简洁，取消了校验和字段

- **常见的过渡技术**：
  - **双栈技术**：设备同时支持IPv4和IPv6协议栈
  - **隧道技术**：将IPv6数据包封装在IPv4数据包中传输
  - **NAT64**：实现IPv6网络访问IPv4资源
  - **DNS64**：配合NAT64使用，合成IPv6地址

## 二、核心协议：路由与交换

这是网工的"看家本领"。

### 1. 二层交换（Switching）

#### VLAN相关

**为什么要划分 VLAN？**

**核心原因：**
1. **安全隔离**：不同部门或用户组之间逻辑隔离，防止未授权访问
2. **控制广播**：限制广播域范围，减少网络拥塞，提高性能
3. **灵活管理**：逻辑分组不受物理位置限制，便于网络调整
4. **节省成本**：充分利用现有设备，减少硬件投资

**典型应用场景：**
- 企业网络：财务、人事、研发等部门隔离
- 校园网络：按楼宇或院系划分不同VLAN
- 数据中心：按业务系统或租户划分网络
- 无线网络：按用户类型（员工/访客）分离流量

**Trunk 链路（802.1Q）的作用**

**核心功能：**

1. **VLAN标签封装**
   - 在以太网帧中插入4字节VLAN标签
   - 标签包含VLAN ID（12位，支持4094个VLAN）
   - 包含优先级信息（3位，支持QoS）
   - 包含CFI位（1位，用于令牌环网络兼容）

2. **多VLAN传输**
   - 单条物理链路承载多个VLAN流量
   - 实现交换机间VLAN信息传递
   - 支持跨交换机的VLAN通信
   - 减少网络布线复杂度

3. **VLAN透传**
   - 保持VLAN信息在传输过程中不变
   - 确保数据包到达目的地时VLAN归属正确
   - 支持VLAN扩展到整个网络
   - 实现统一的VLAN管理策略

**技术实现细节：**

```
标准以太网帧格式：
[目的MAC][源MAC][类型/长度][数据][FCS]

802.1Q tagged帧格式：
[目的MAC][源MAC][8100][VLAN标签][类型/长度][数据][FCS]

VLAN标签结构：
TPID(16位) = 0x8100（标识802.1Q帧）
TCI(16位) = PCP(3位) + CFI(1位) + VID(12位)
```

**配置要点：**
- Trunk端口允许多个VLAN通过
- Access端口只属于单一VLAN
- Native VLAN用于传输未标记流量
- 需要在互联设备上配置相同的VLAN信息
