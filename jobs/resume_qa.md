**Q: 上线了吗？**

**Q: 性能数据？**

**A**: 在仿真环境下验证了逻辑可行性。

**补充说明**: 由于未部署物理机，具体的硬件转发性能受限于芯片型号，我主要负责控制面逻辑的稳定性。

**Q：在 SONiC 中，为什么要单独设计 dot1xcfgd 和 dot1xsyncd 两个守护进程？它们的交互流程是怎样的？**

**A**: 这是为了遵循 SONiC 的解耦设计哲学。dot1xcfgd 负责监听用户态的配置下发，将其转化为结构化的数据写入 Redis 的 CONFIG_DB 。

dot1xsyncd 则负责订阅状态变化，将认证通过后的授权信息（如FDB、ACL规则）下发至硬件，并将实时的流量统计和上线状态反馈回 APP_DB 。这种异步订阅机制保证了管理面和转发面的独立性，即便单个容器重启，也不会影响已在线的用户流量 。

**Q：简历提到支撑‘单台设备万级用户稳定在线’，你如何解决高并发认证时的 CPU 压力或状态机死锁问题？**

**A**：核心方案是异步处理与流量分级。我们在 dot1xsyncd 中实现了批量处理机制，避免频繁读写 Redis 造成的系统开销。

同时，针对 hostapd 的事件循环进行了优化，引入了基于流量/时长的平滑刷新逻辑，避免所有用户在同一时刻进行状态重认证，从而将 CPU 峰值负载降低

**Q：你的文件扫描模块是如何基于命名空间和特性目录进行优化的？比传统的全盘扫描快在哪？**

**A**：“我们放弃了传统的暴力遍历。通过解析命名空间（Namespace）和特性目录参数，我们将扫描范围限制在特定的逻辑路径下 。”

“这种元数据驱动的扫描方式，通过直接读取特定命名空间下的索引表项，避免了无关目录的深层递归。在千万级文件的规模下，将扫描初筛的时间缩短了显著比例，为后续数据在不同存储介质间的自动迁移提供了及时的触发信号 。”

**Q：你既做过网络开发，也支援过存储开发，你觉得这两个领域的系统设计有什么异同？**

**A**：“相同点在于对高性能和资源利用率的极致追求。无论是处理万级用户认证 ，还是千万级小文件存储 ，核心都在于减少锁竞争、优化数据结构和异步化处理。”

“不同点在于侧重点。网络开发（如 SONiC/FRR）更强调实时性和协议一致性，要在毫秒级完成路由收敛 ；而存储开发更强调数据持久性与一致性，尤其是在处理异构分级存储时，如何保证迁移过程中的数据安全是重中之重 。”

## 其他项目

### SSH 项目

#### 1. 项目背景与目标

在 SONiC 系统中实现 SSH 安全配置的持久化管理。目标是让用户能通过命令行（CLI）动态调整 SSH 参数（如使能、端口、访问权限），并确保配置在系统重启后依然生效。

#### 2. 核心架构设计 (The "Why" and "How")

你可以这样向面试官介绍你的设计思路：

**数据流向：** 我采用了 SONiC 标准的数据驱动架构。用户在 CLI 输入命令后，配置并不直接修改文件，而是首先写入 ConfigDB（Redis 数据库）中的 SSH 表。

**解耦机制：** 为了实现配置的实时生效，我利用现有的 hostcfgd。该进程利用 Redis 的 Pub/Sub 机制订阅 ConfigDB 的变化。

**落地执行：** 一旦检测到 SSH 配置更新，hostcfgd 会根据数据库内容重新渲染 SSH 配置文件模板，并调用 `systemctl reload sshd` 使配置生效（reload 可以避免连接中断）。

#### 3. 方案的优越性（面试加分点）

当面试官问"为什么要这么设计"时，你可以给出以下三个专业理由：

**唯一真理源 (Single Source of Truth)：** 所有的管理接口（CLI、REST API、gNMI）都统一操作 ConfigDB，避免了多头管理导致的配置冲突。

**原生持久化：** SONiC 的 `config_db.json` 会自动备份数据库内容，这样即便设备断电重启，SSH 配置也能自动恢复，无需额外写复杂的脚本去同步文件。

**系统解耦：** 配置逻辑与底层实现分离。如果未来 SSH 服务更换了实现方式，我们只需要修改 Daemon 里的文件渲染逻辑，而不需要改动上层 CLI 或数据库结构。

#### 4. 面试常见追问模拟

**面试官：** "如果写入数据库成功了，但修改配置文件失败了（比如磁盘满了），你如何处理这种不一致？"

**你的回答思路：**

这是一个非常深入的问题。在设计时，Daemon 进程会捕获文件写入和重启服务的返回值。如果失败，我会：

- **记录日志**：通过 syslog 报错以便运维排查。
- **状态回滚/反馈**：在更完善的机制中，可以引入 STATE_DB 来反馈当前配置的"运行状态"（Applied vs Failed），让管理员知道配置虽然在数据库里，但实际并未生效。

**面试官：** "如果用户手动改了文件怎么办？"

**你的回答思路：**

"在 SONiC 的设计哲学中，ConfigDB 是系统的唯一事实源。为了防止配置漂移（Configuration Drift），我采取了'强制同步'的策略：

- **初始化同步**：每当 hostcfgd 启动或重启时，它会首先从 Redis 读取最新的 SSH 配置，并通过 Jinja2 模板全量覆盖本地的 `/etc/ssh/sshd_config`。
- **运行期收敛**：这种设计确保了即便有人在系统底层进行了临时的手动修改，只要服务重启或配置更新，系统最终都会自动收敛回数据库定义的'标准状态'。
- **管理闭环**：这迫使运维人员必须通过北向接口（CLI/API）来修改配置，从而保证了配置的可审计性和一致性。"

## 面试实战案例与应对策略

### 案例：面试官认为"难点不够难"

**场景描述**：我昨天拿这个简历去面试，面试官让我介绍我项目中的难点和亮点，我拿第一个项目举例子，说 dot1xcfgd 这个守护进程虽然使用 python 写的，但是对于当时刚入职的我来说，从 0 到 1，然后测试维护，还是挺难的；第二点是容器内新增 dot1xsyncd 进程，负责和 hostapd 和 redis 数据库交互，难点在于和 hostapd 进程之间定义消息类型，以及如何通信。面试官听了后，觉得这些都不算难点，都是可以参考的，我应该怎么准备后面面试中类似的问题。

**问题分析**：面试官之所以觉得"不难"，是因为你描述的侧重点在于**"个人的上手过程"**（从 0 到 1、写代码、定义消息），而面试官期待的是**"技术的复杂深度"**（高并发、状态一致性、系统瓶颈、异常边界处理）。

对于像你这样有 3.5 年经验、且有 Committer 身份的开发人员，你需要把"工程实现"提升到**"架构设计与系统可靠性"**的高度。以下是针对你第一个项目的三个"高级"难点准备方向：

### 1. 难点一：分布式状态的一致性与同步（深挖 `dot1xsyncd`）

**原说法**：定义消息类型和通信很难。

**高手说法**：**"如何确保协议状态机、Redis 数据库与硬件转发表项（FDB/ACL）在各种极端情况下的强一致性。"**

- **技术细节**：在 SONiC 这种基于数据库驱动的架构中，`dot1x` 容器是用户态的，而 FDB/ACL 是硬件底层的。
- **你可以这样讲**：
  - "难点不在于定义消息，而在于**异步系统的同步效率**。当万级用户并发上线时，`hostapd` 的协议状态、Redis 中的 `app_db` 以及底层的物理表项之间存在同步延迟。如果处理不好，会出现'协议已认证但流量不通'或'用户已离线但表项未删除'的黑洞风险。"
  - **解决方案**：你通过 `dot1xsyncd` 引入了**状态重试与校验机制**，确保每一次表项写入都有反馈闭环，并优化了数据库的批处理逻辑以支撑高并发场景。

### 2. 难点二：高可用性与"热重启"下的用户无感恢复

**原说法**：测试和维护挺难。

**高手说法**：**"设计并实现了一套'进程重启数据不丢失'的恢复机制，保障系统高可用。"**

- **技术细节**：如果 `dot1x` 容器崩溃重启，内存里的用户状态全丢了，如果直接清空硬件表项，会导致所有在线用户掉线。
- **你可以这样讲**：
  - "为了实现电信级的可靠性，我设计了**用户数据恢复逻辑**。在 `dot1x` 容器重启时，进程会首先通过 `dot1xsyncd` 从 Redis `state_db` 或硬件 FDB 中反向'回捞'当前在线用户信息，并重新注入到 `hostapd` 的动态状态机中。"
  - **难点**：难在如何处理"恢复期间"产生的新报文。你需要协调状态机的初始化顺序，避免新旧状态冲突。

### 3. 难点三：复杂业务场景下的触发逻辑（如 PVID 动态感知）

**原说法**：写 Python 脚本配置功能很难。

**高手说法**：**"解决了动态网络拓扑变化下的协议联动难题，例如接口 PVID 变化导致的状态机非法转换。"**

- **技术细节**：接口配置变化（如 PVID 变更）会直接影响 802.1X 的认证 VLAN 环境。
- **你可以这样讲**：
  - "在开发过程中，我发现当用户已上线但管理员修改了接口 PVID 时，现有的开源 `hostapd` 无法感知这种二层配置变化。我通过监听 SONiC 的 `CONFIG_DB` 变化，实现了**接口事件实时通知机制**，强制触发用户重新认证或迁移表项。"
  - **价值**：这证明你具备**系统级联动**的眼界，而不仅仅是盯着自己那一块 Python 脚本。

### 准备后续面试的万能公式（STAR 法则技术版）

4. **最终产出 (Result)**：用数字说话。（例如：顺利合入主干，支持了带 VLAN 认证等多种复杂组网场景，提升了系统稳定性）。
