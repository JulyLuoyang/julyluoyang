## 一、进程与线程

### 1.1 进程 vs 线程
| 对比项 | 进程 | 线程 |
|--------|------|------|
| 资源拥有 | 独立地址空间、文件描述符等 | 共享进程资源，仅有栈/寄存器等私有数据 |
| 切换开销 | 大（需切换页表、TLB 刷新） | 小（同进程内只需切换栈和寄存器） |
| 通信方式 | IPC：管道、消息队列、共享内存、socket | 直接读写共享变量 + 同步机制 |
| 健壮性 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能导致整个进程退出 |

🔥 **追问**：为什么线程切换比进程切换快？
```
答：线程切换不需要切换页表（mm_struct 相同），TLB 无需刷新，
    地址空间不变，缓存命中率更高。
```

### 1.2 进程状态与转换
```
新建 → 就绪 ⇄ 运行 → 阻塞 → 就绪 → 终止
              ↑
           时间片到/被抢占
```

🔥 **Linux 特有**：`TASK_INTERRUPTIBLE` / `TASK_UNINTERRUPTIBLE` / `TASK_STOPPED`
- `UNINTERRUPTIBLE` 状态进程不能被 `kill -9` 杀死，常用于等待关键 I/O（如磁盘），避免数据不一致

### 1.3 进程间通信（IPC）方式对比
| 方式 | 特点 | 适用场景 |
|------|------|----------|
| 管道 | 半双工，亲缘进程 | shell 命令链 `|` |
| 命名管道 | 任意进程，基于文件系统 | 无亲缘关系简单通信 |
| 消息队列 | 内核维护队列，带优先级 | 任务分发、异步解耦 |
| 共享内存 🔥 | 最快，需配合同步机制 | 高频大数据交换（如 SONiC SwSS 与 app_db） |
| 信号 | 异步通知，携带信息少 | 进程控制（kill、SIGUSR1） |
| 信号量 | 计数器，控制资源访问 | 多进程同步 |
| Socket | 跨机器，通用 | 分布式系统、C/S 架构 |

> 💡 结合简历：SONiC 中 `syncd` 与 `swss` 通过 Redis（共享内存+网络）通信，本质是 IPC 的工程实践

## 二、进程调度

### 2.1 调度算法演进
```
FCFS → SJF → 优先级调度 → 时间片轮转 → 多级反馈队列（MLFQ）🔥
```

### 2.2 Linux CFS（完全公平调度器）核心思想
- **目标**：让每个任务在一段时间内获得公平的 CPU 时间
- **关键数据结构**：红黑树（按 `vruntime` 排序）
- **vruntime** = 实际运行时间 × 权重（nice 值决定）
- **调度流程**：
  1. 选红黑树最左节点（vruntime 最小）执行
  2. 执行后更新 vruntime，重新插入树中
  3. 新任务/唤醒任务：vruntime = min(当前最小值, 全局平均值)

🔥 **追问**：实时进程怎么调度？
```
答：Linux 实时进程（SCHED_FIFO / SCHED_RR）优先级高于 CFS 进程，
    采用优先级队列，高优先级实时任务永远优先执行。
```

### 2.3 上下文切换开销优化（高频！）
```c
// 减少切换的实践：
1. 绑定 CPU 亲和性（sched_setaffinity）→ 减少缓存失效
2. 使用线程池/协程 → 避免频繁创建销毁
3. 减少锁竞争 → 用无锁结构/细粒度锁
4. 网络场景：NAPI + 中断合并 → 减少软中断触发频率
```

## 三、内存管理

### 3.1 虚拟内存核心价值
```
✅ 隔离：进程间地址空间独立，安全
✅ 抽象：连续虚拟地址映射非连续物理页
✅ 扩展：使用磁盘做 swap，突破物理内存限制
✅ 共享：多个进程映射同一物理页（如库文件）
```

### 3.2 页表与 TLB
- **多级页表**：x86_64 通常 4 级（PML4→PDPT→PD→PT），减少页表内存占用
- **TLB（Translation Lookaside Buffer）**：页表项缓存，命中则无需查页表
- 🔥 **大页（Huge Page）**：2MB/1GB 页，减少 TLB miss，适合 Redis/DPDK 等高性能场景

### 3.3 内存分配机制
```
用户态：malloc → brk(小内存) / mmap(大内存)
内核态：__get_free_pages(连续物理) / kmalloc(带缓存) / vmalloc(连续虚拟)
```

🔥 **slab 分配器**：内核缓存常用对象（如 task_struct、socket），避免频繁分配释放

### 3.4 零拷贝技术（网络开发必问！）
```c
// 传统拷贝：磁盘→内核缓冲区→用户缓冲区→socket 缓冲区→网卡（4 次拷贝+2 次上下文切换）

// 零拷贝方案：
1. mmap + write：减少 1 次用户态拷贝
2. sendfile 🔥：Linux 2.1+，内核直接拷贝，0 次用户态拷贝
3. splice：管道 + sendfile，适合代理场景
4. DMA + SG-DMA：网卡直接访问用户内存（需页锁定）

// 应用：Nginx 静态文件、SONiC 镜像下发、DPDK 包处理
```

## 四、并发与同步

### 4.1 常见同步原语对比
| 原语 | 特点 | 适用场景 |
|------|------|----------|
| 互斥锁（mutex） | 独占访问，可重入（pthread_mutex_t 可配置） | 临界区保护 |
| 自旋锁 🔥 | 忙等待，不睡眠，适合短临界区 | 中断上下文、锁持有时间极短 |
| 读写锁 | 读共享、写独占 | 读多写少（如配置缓存） |
| 条件变量 | 配合 mutex，实现等待/通知 | 生产者-消费者 |
| 信号量 | 计数型，控制资源数量 | 连接池、限流 |
| RCU 🔥 | 读拷贝更新，读无锁、写延迟释放 | Linux 内核路由表、网络栈 |

### 4.2 死锁四要素与预防
```
必要条件：互斥 + 请求保持 + 不可剥夺 + 循环等待
预防策略：
1. 破坏请求保持：一次性申请所有资源
2. 破坏不可剥夺：允许抢占（难实现）
3. 破坏循环等待：按序申请资源（最常用）
4. 银行家算法：避免不安全状态（理论为主）
```

🔥 **调试技巧**：
```bash
# 查看进程锁等待
cat /proc/<pid>/stack
# GDB 附加进程查锁
gdb -p <pid> → info threads → thread apply all bt
# 用户态：pthread_mutex 可设置错误检查类型
```

### 4.3 原子操作与内存屏障
```c
// 原子操作（无需锁）
atomic_add(&val, 1);  // 底层：lock addl [mem], imm

// 内存屏障（防止指令重排）
smp_mb();   // 全屏障
smp_wmb();  // 写屏障
smp_rmb();  // 读屏障

// 应用场景：无锁队列、引用计数、标志位
```

## 五、文件系统与 I/O

### 5.1 Linux 文件描述符与 inode
```
fd（进程级）→ 指向 file 对象（内核级）→ 指向 inode（磁盘元数据）
✅ 同一个文件可被多个 fd 打开，共享 offset（除非 O_APPEND）
✅ hard link：多个目录项指向同一 inode
✅ soft link：独立文件，存储目标路径
```

### 5.2 I/O 模型对比（网络开发核心！）
| 模型 | 阻塞/非阻塞 | 复用方式 | 特点 |
|------|------------|----------|------|
| blocking I/O | 阻塞 | - | 简单，但并发差 |
| non-blocking I/O | 非阻塞 | 轮询 | CPU 空转，不推荐 |
| I/O 多路复用 🔥 | 非阻塞 | select/poll/epoll | 高并发基石 |
| signal-driven I/O | 异步通知 | sigio | 少用，信号处理复杂 |
| AIO 🔥 | 真正异步 | io_submit/io_getevents | Linux 实现不完善，常用协程替代 |

### 5.3 epoll 原理（必考！）
```c
// 优势 vs select/poll：
1. 红黑树管理监听集合：O(1) 增删改
2. 就绪链表：只返回就绪 fd，避免全量扫描
3. mmap 减少用户/内核拷贝（部分实现）

// 两种触发模式：
- LT（水平触发，默认）：就绪状态持续通知，安全可靠
- ET（边缘触发，高效）：仅状态变化时通知，需一次读/写完 + 非阻塞 + EAGAIN 判断

// 实践建议：
✅ ET 模式 + 非阻塞 socket + while(read) 循环
✅ 用 epoll_ctl(EPOLL_CTL_MOD) 动态调整监听事件
```

### 5.4 页缓存与直接 I/O
```
Page Cache：内核缓存文件页，加速重复读/延迟写
✅ 优势：提升吞吐，合并写操作
❌ 问题：掉电丢失，大文件污染缓存

Direct I/O（O_DIRECT）：
- 绕过 Page Cache，用户缓冲区 ↔ 磁盘
- 要求：内存对齐、大小对齐
- 场景：数据库（MySQL InnoDB）、分布式存储

💡 结合简历：你做的异构存储分级，可讨论 Page Cache 策略 + Direct I/O 的权衡
```

## 六、Linux 内核与系统编程

### 6.1 系统调用流程
```
用户态调用 libc → trap 指令（int 0x80 / syscall）→ 
CPU 切换内核态 → 查 syscall 表 → 执行内核函数 → 
返回用户态（保存/恢复上下文）

🔥 优化：vDSO（virtual dynamic shared object）
   部分系统调用（如 gettimeofday）直接在用户态完成，避免 trap
```

### 6.2 中断与软中断（网络驱动重点！）
```
硬中断：网卡收到包 → 触发 IRQ → 关中断 → 快速拷贝包到 ring buffer → 触发软中断
软中断（NET_RX_SOFTIRQ）：批量处理协议栈（IP/TCP/应用）

🔥 NAPI 机制：
- 高负载时：关闭硬中断，改为轮询 + 软中断，减少中断风暴
- 低负载时：恢复中断模式，降低延迟

💡 结合简历：你熟悉网络驱动，可深入讲 NAPI + GRO（Generic Receive Offload）
```

### 6.3 容器底层技术（结合 Docker/SONiC）
```
1. Namespace 🔥：资源隔离
   - pid/net/mnt/uts/ipc/user → 实现"独立系统视图"
2. Cgroups 🔥：资源限制
   - cpu/memory/blkio → 防止容器资源争抢
3. UnionFS：分层镜像（overlay2）
4. bridge/veth：容器网络（可延伸讲 SONiC 的 network namespace 应用）

🔥 追问：容器和虚拟机区别？
答：容器共享内核，轻量但隔离性弱；VM 有完整 Guest OS，隔离强但开销大
```

#### 一、 核心概念（必考基础）
容器 vs 虚拟机 (VM)：

VM：包含完整的操作系统，有独立的 Hypervisor 层，资源占用大，启动慢。

Docker：共享宿主机内核，利用 Namespace 和 Cgroups 实现隔离，轻量级，秒级启动。

三个基本组件：

Image (镜像)：只读的模板，包含运行环境。

Container (容器)：镜像的运行实例（可读写层）。

Repository (仓库)：集中存放镜像的地方（如 Docker Hub）。

#### 二、 底层原理（区分大厂面试的关键）
这是面试中最能体现深度的地方，务必背熟这三个词：

Namespace (命名空间)：实现资源隔离（如 PID、Net、IPC、Mount 等），让容器觉得自己在独立的操作系统里。

Cgroups (控制组)：实现资源限制（如限制容器最多使用 2GB 内存、50% CPU），防止某个容器榨干宿主机。

UnionFS (联合文件系统)：实现分层存储。镜像是由一层层只读层组成的，通过“写时复制”（Copy-on-Write）技术，多个容器可以共享镜像层，节省空间。

#### 三、 网络与存储
网络模式 (Network)：

Bridge (默认)：通过虚拟网桥 docker0 互联。

Host：容器直接使用宿主机 IP 和端口（无隔离，性能最高）。

None：不配置网络，安全性最高。

Overlay：用于跨主机的容器通信（Swarm/K8s 常用）。

持久化 (Storage)：

Volume (卷)：Docker 管理，存储在宿主机特定位置，最推荐。

Bind Mount (绑定挂载)：挂载宿主机任意路径，适合开发调试。

#### 四、 Dockerfile 优化（实战经验）
面试官常问：“如何减小 Docker 镜像体积？”

使用精简基础镜像（如 alpine 或 distroless）。

合并 RUN 指令（减少镜像层数）。

使用多阶段构建 (Multi-stage builds)（只把编译后的产物带入最终镜像）。

利用 .dockerignore 排除不必要的文件。

#### 五、 常见高频面试题
CMD 和 ENTRYPOINT 的区别？

CMD 指定默认运行命令，会被 docker run 后面的参数覆盖。

ENTRYPOINT 不会被覆盖，适合把容器当作可执行程序使用。

如何进入一个正在运行的容器？

docker exec -it <container_id> /bin/bash（最常用，开启新进程）。

docker attach（接入原进程，退出会导致容器停止）。

容器退出后，数据会丢失吗？

不会。除非 docker rm 删除了容器。但建议重要数据一定要挂载 Volume。

### 6.4 性能调优常用命令
```bash
# CPU

perf top           # 热点函数采样
pidstat -u 1       # 进程级 CPU 统计

```

## 七、高频场景题（结合你的经历）

### 7.1 万级用户在线，如何保证认证服务不卡顿？
```
✅ 多进程/线程池：避免单点阻塞，利用多核
✅ 异步非阻塞：epoll + 状态机处理认证流程
✅ 无锁队列：认证请求入队，worker 线程消费
✅ 内存池：预分配认证上下文，避免频繁 malloc
✅ 限流降级：突发流量时拒绝非关键请求
✅ 监控告警：埋点统计认证耗时/P99 延迟

💡 结合简历：你做的 dot1xsyncd 可讲"Redis 管道批量操作 + 连接池复用"
```

### 7.2 ISIS 协议收敛慢，如何从 OS 层优化？
```
✅ 绑定 CPU 亲和性：协议进程固定核，减少缓存失效
✅ 实时优先级：SCHED_FIFO 保证协议计算不被抢占
✅ 大页内存：路由表用 HugePage，减少 TLB miss
✅ 零拷贝：协议报文用 sendfile/splice 转发
✅ 中断优化：网卡多队列 + RSS，分散协议处理压力

💡 结合简历：你修复过 LFA Tiebreaker 缺陷，可讲"配置热更新时如何避免路由震荡"
```

### 7.3 分布式存储中，小文件读写性能瓶颈在哪？
```
✅ 元数据瓶颈：每个文件独立 inode，mds 压力大 → 解决方案：元数据缓存 + 批量提交
✅ 磁盘寻道：HDD 随机写慢 → 解决方案：小文件合并（WAL/Log-Structured）
✅ 网络开销：RPC 频繁 → 解决方案：批处理 + 零拷贝 + RDMA
✅ 锁竞争：全局锁 → 解决方案：分片锁/无锁哈希

💡 结合简历：你做的异构分级，可讲"热点文件自动迁移到 NVMe 的策略设计"
```
